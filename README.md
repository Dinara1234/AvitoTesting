## 1. Введение
**Базовый URL:** `https://qa-internship.avito.com`  
**Ресурсы API:**
- `POST /api/1/item` – создать объявление
- `GET /api/1/item/{id}` – получить объявление по ID
- `GET /api/1/{sellerID}/item` – получить все объявления продавца
- `GET /api/1/statistic/{id}` – получить статистику по объявлению

**Особенности:**
- При создании генерируется уникальный `id` (строковый).
- Поля `name`, `price`, `sellerId`, `statistics` могут повторяться у разных объявлений.
- `sellerId` – целое число, рекомендуется диапазон `111111–999999`.

## 3. Корнер-кейсы и идемпотентность

| TC ID | Описание | Шаги | Ожидание |
|-------|----------|------|-----------|
| CORNER-01 | Одновременное создание двух объявлений с одинаковыми данными (конкурентность) | Два параллельных запроса POST с одинаковым телом | Оба успешны, возвращают разные `id`. Нет ошибок 5xx |
| CORNER-02 | Идемпотентность GET-запросов | Повторить GET /api/1/item/{id} 10 раз | Каждый раз одинаковый ответ (200, данные те же) |
| CORNER-03 | Создание объявления с price = 0 и отрицательными значениями статистики | `{"price": 0, "statistics": {"likes": -1, "viewCount": -1, "contacts": -1}}` | Код 400, отрицательная статистика не допускается |
| CORNER-04 | Создание объявления с очень большими значениями статистики (10^9) | `likes: 999999999` | Код 200 или 400 – проверка лимитов |
| CORNER-05 | Использование спецсимволов в name: `!@#$%^&*()` | Создать объявление с таким name | Код 200, данные сохраняются корректно (без XSS) |
| CORNER-06 | Очень длинный sellerID (например, 999999999999) | Создать объявление | Если API поддерживает 64-битные числа, то 200; если нет – 400 |
| CORNER-07 | Запрос статистики для id, который является числом, но не существует | id = "1234567890" (не создавали) | Код 404 |
| CORNER-08 | Передача в POST пустого тела | – | Код 400 |

## 4. Нефункциональные проверки

| TC ID | Тип | Описание | Ожидание |
|-------|-----|----------|-----------|
| NF-01 | Производительность (нагрузка) | 1000 последовательных запросов POST за 30 секунд | Среднее время ответа < 500 мс, без ошибок 5xx |
| NF-02 | Производительность (параллелизм) | 50 одновременных GET /api/1/{sellerID}/item для продавца со 100 объявлениями | Все ответы 200, время < 1 с |
| NF-03 | Безопасность (SQL-инъекция) | В поле name передать `"'; DROP TABLE items; --"` | Код 200 (данные экранированы), таблицы не удалены |
| NF-04 | Безопасность (XSS) | В name передать `<script>alert(1)</script>` | При GET ответ содержит экранированный текст, браузер не исполняет скрипт |
| NF-05 | Устойчивость (длительное использование) | В течение 1 часа каждые 5 секунд создавать и удалять объявления | Нет утечек памяти, ответы стабильны |
| NF-06 | Обработка больших объёмов данных | Запросить все объявления продавца, у которого 10 000 объявлений | Ответ не должен превышать лимит (например, 10 MB). Либо реализована пагинация |
| NF-07 | Версионирование (API v1 vs v2) | Сравнить ответы GET /api/1/statistic/{id} и GET /api/2/statistic/{id} | Формат может отличаться (в коллекции /api/2 возвращает массив). Оба должны работать стабильно |
